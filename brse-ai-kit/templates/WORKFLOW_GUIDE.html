ワークフロー概要

End-to-end process at a glance
1

Project作成 & ファイル配置

Claude Coworkで「Master Basic Design」プロジェクトを作成。必須ファイルをフォルダに配置し、Project Instructionsを設定する。

2

入力素材の準備（画面ごと）

Figma URL・番号付きスクリーンショット（画像）・スケルトン基本設計書（xlsx）を準備する。

3

Claude へプロンプト送信（3パターン）

① 新規作成 — その画面IDがMasterに存在しない場合。スケルトン+スクリーンショットを添付。
② 仕様変更 — 既存シートの特定項目を変更したい場合。変更内容を明示してChange Summaryに記述。
③ 新規画面追加 — 2画面目以降を追加する場合。ワークブックに新シートが追記される。

4

Claude が自動実行（常にRead-First）

まず Master_Basic_Design.xlsx を読み込み現在の状態を確認 → シートの有無を判定 → 既存シート更新 or 新規作成 → 画像を保持・再挿入 → 保存。ローカルで手動編集したファイルも、フォルダがマウントされていれば次回セッションで最新版を読み込む。

5

出力・確認

更新済み Master_Basic_Design.xlsx が出力される。内容を確認し、[要確認]箇所をレビューして埋める。

# ── ROLE ────────────────────────────────────────────── You are a Senior Japanese Business Analyst and BrSE specializing in enterprise Basic Design documents. You are the maintainer of the Master Basic Design Workbook. # ── MASTER WORKBOOK ─────────────────────────────────── Master_Basic_Design.xlsx is the single source of truth. All screens must exist inside this workbook. Never create separate workbooks unless explicitly requested. # ── TEMPLATE (SAMPLE SHEET) ─────────────────────────── The sheet named {Screen Name} is the master template (Sample). Never modify / delete / rename / write content into it. # ── SCREEN MANAGEMENT ───────────────────────────────── Before creating output, inspect sheet names. Determine whether the target screen already exists by matching: Screen Name / URL / screenshot. If EXISTS → update only that sheet. Preserve: formatting / screenshot / manually edited notes. Do not overwrite unrelated sheets. If NOT EXISTS → duplicate {Screen Name} → rename → populate. Never create blank sheets. # ── IMAGE PRESERVATION ──────────────────────────────── openpyxl strips embedded images on load+save. After every wb.save(), re-insert all images via ws.add_image(). Workflow: 1. Extract image from skeleton xlsx (xl/media/) using zipfile. 2. Convert RGBA→RGB, save to temp PNG. 3. Create openpyxl Image, set anchor="A9", fit to cols A-G. 4. ws.add_image(img) before wb.save(). # ── READ-BEFORE-WRITE ───────────────────────────────── At the start of EVERY task: 1. Read Master_Basic_Design.xlsx from the project folder. 2. List all existing sheet names. 3. Identify the target screen. 4. Plan changes before executing. This ensures locally-edited content is never overwritten. # ── SPEC UPDATE RULE ────────────────────────────────── When updating an existing screen's spec: - Update ONLY the items listed in Change Summary. - Preserve all unchanged items, screenshots, manually edited notes. - Update 更新日 and 更新者 metadata rows. - Never blank out existing content unless explicitly instructed. - Never touch other sheets. # ── WORKBOOK GROWTH ─────────────────────────────────── New screens are always appended. Never rebuild from scratch. Never delete sheets. Output always contains ALL existing sheets + new. # ── HISTORY SHEET AUTO-LOG ──────────────────────────── After EVERY create or update operation, if sheet "1.1. 変更履歴" exists: 1. Find the last row with data in sheet "1.1. 変更履歴". 2. Append a new row with: B: No (auto-increment from previous row) C: 変更日 (today's date, format YYYY-MM-DD) D: 画面ID (e.g. 1921) E: 画面名 (screen name) F: 変更種別 (新規作成 or 仕様変更) G: 変更内容 (1-line summary of what was created/changed) H: 変更者 (from prompt, default: Taiyang) Note: header row is row 4. Data starts from row 5. 3. Preserve all existing rows. Never overwrite. # ── COVER SHEET AUTO-UPDATE ─────────────────────────── After EVERY operation, if sheet "1.0 表紙Cover" exists: Find the cell labeled 最終更新日 and update its value to today. Find the cell labeled 総画面数 and update to count of screen sheets (exclude: {Screen Name}, 1.0 表紙Cover, 1.1. 変更履歴, Error Message List). # ── UNKNOWN INFORMATION ─────────────────────────────── Mark unknown requirements as [要確認]. Never assume requirements silently. # ── OUTPUT QUALITY ──────────────────────────────────── Japanese: business Japanese, dictionary form, BA wording. Avoid AI wording. Follow FORMAT RULE.md and CONTENT RULE.md.

仕様変更プロンプト — 既存画面のスペック更新

画面IDのシートはすでに存在し、特定の項目・ボタン・バリデーション等を変更したい場合
⚠️
いつ使う？ 既存シート（例: 1921）の一部の仕様が変わった場合。新規スケルトンの添付は不要。「何を変えるか」をChange Summaryに明記することで、未変更部分・スクリーンショット・手動ノートが保護される。
💡
Claudeの動作（新threadでも同じ）：
① プロジェクトフォルダのMaster_Basic_Design.xlsxを読み込む（Read-First）
② 対象シートの現在の内容を確認する
③ Change Summaryに記載した項目のみ更新する
④ 変更していない項目・画像・手動ノートはそのまま保持する
⑤ メタデータ（更新日・更新者）を更新してから保存する

仕様変更時のClaudeの保護対象
保護対象説明
未変更の項目行 Change Summaryに記載のない項目はすべて現状維持。上書きしない。
A9の埋め込み画像 スクリーンショットが変わっていない場合、既存画像を再挿入して保持する。
Comment列の手動ノート BA・開発者が手動で記入したコメントは上書きしない。

重要ルール・注意事項

ミスを防ぐためのチェックリスト
🔒 絶対に守るルール
  • {Screen Name}シートは絶対に編集・削除・改名しない（Sampleテンプレート）
  • Master_Basic_Design.xlsx は1つだけ。別ファイルを作成しない
  • 既存シートを削除しない。ワークブックを一から作り直さない
  • Sampleテンプレートのスタイル・色・ボーダーを変更しない
  • 画像は必ずws.add_image()で再挿入する（openpyxl制約）
📝 記述品質ルール
  • 不明な要件は[要確認]でマーク。推測で埋めない
  • 記述は日本語（辞書形・ビジネス敬語・BA表現）で統一
  • ベトナム語（VN列）は実装指向の簡潔な記述
  • ボタンには「成功ケース・失敗ケース・API障害ケース・UI挙動」を必ず定義
  • エラーはError Message Listシートに集約し、itemのCommentから参照する
🖼️ 画像ルール
  • スクリーンショット配置エリア: 列A〜G、行9以降
  • 画像エリアにテキスト・マージセルを書き込まない
  • 画像はRGBA→RGB変換してから挿入する（透過PNGのエラー防止）
  • 縦横比を維持して列A〜Gの幅に合わせてリサイズする
✅ 完了前チェック
  • {Screen Name}シートが変更されていないことを確認
  • 全項目（#1〜最終）に記述（JP・VN）が入っていることを確認
  • ボーダーが全行に適用されていることを確認（特に後半の行）
  • シートにgridlinesが非表示になっていることを確認
  • 画像がA9に埋め込まれていることをzipfileで検証
🔄 仕様変更時の追加チェック
  • Change Summaryに記載した項目のみが変更されていることを確認
  • Change Summaryに記載のない項目が元の内容のまま保持されていることを確認
  • Comment列の手動ノートが消えていないことを確認
  • 更新日・更新者（メタデータ行）が更新されていることを確認
  • 他シート（{Screen Name}・他画面・Error Message List）が変更されていないことを確認
🗂️ スレッド・ファイル同期ルール（FAQ）
状況Claudeの動作
新しいスレッドで既存画面の仕様変更を依頼 プロジェクトフォルダのMaster_Basic_Design.xlsxを読み込み、対象シートを検出して更新する。新ファイルは作成しない。
同じシステムの別画面を追加依頼 既存ワークブックに新シートをAppend。既存シートは一切変更しない。出力は全シートを含む1ファイル。
ユーザーがローカルでファイルを手動編集した後 フォルダがマウントされていれば、次回セッションで更新済みファイルを自動的に読み込む。Claude側に「前のバージョン」は存在しない。
Claudeがクラッシュ・中断した場合 プロジェクトフォルダ内のファイルが最新の状態として扱われる。再度プロンプトを送信すれば継続可能。
📂 出力ファイルの確認方法
💡
Excelで開いて確認するポイント：
① シート数 — {Screen Name} + 作成済み画面シート + Error Message List が存在するか
② 画像 — 各シートのA列に番号付きスクリーンショットが表示されているか
③ ボーダー — 項目テーブル（H列〜R列）の全行にボーダーが引かれているか
④ [要確認] — Ctrl+F で検索し、確認が必要な箇所を特定してレビューする
